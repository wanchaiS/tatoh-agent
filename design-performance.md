# Search Results — Image Performance

## Context

The room search results (map explorer and any card-list surface) are slow because the image pipeline is naive. A focused audit found:

- **Backend generates exactly one thumbnail size**: `600×400 JPEG @ 85%` via PIL, written to `/static/photos/rooms/{room_id}/thumbnails/{filename}` (`agent_api/api/rooms/photo_router.py`). No additional sizes, no WebP.
- **Collapsed thumbnails are over-fetched ~2.5×**: the card renders at `160×130` CSS pixels but downloads the 600×400 thumbnail.
- **Carousel pre-fetches every photo even when the card is collapsed**: `RoomCardVertical.tsx` always mounts `<img>` tags inside the `grid-rows-[0fr]` collapse container, so the browser fetches them regardless of visibility. For 16 rooms × ~10 photos = **~160 full-resolution images** downloaded before the user expands anything.
- **Carousel uses `url` (full-res), not `thumbnail_url`**: full-resolution originals (up to 5 MB) get pulled into a ~300-wide viewport slot.
- **No `loading="lazy"` anywhere**. No `srcset`/`sizes`. No `width`/`height` attributes → avoidable CLS. Shimmer skeleton is the only placeholder.
- **No CDN / image proxy** — FastAPI `StaticFiles` mount only.

The user's instinct ("we need different sizes of pictures") is correct and is the structural fix. This is a pre-production project, so we build the final shape on day one — no backward-compat aliases, no rollout shims, no backfill.

## Plan

One coordinated change touching the photo pipeline end-to-end.

### 1. Backend — multi-size thumbnails

In `agent_api/api/rooms/photo_router.py`, extend the upload handler to generate **three** JPEG thumbnails per upload, written to size-partitioned directories:

```
/static/photos/rooms/{room_id}/thumbnails/240/{filename}
/static/photos/rooms/{room_id}/thumbnails/480/{filename}
/static/photos/rooms/{room_id}/thumbnails/960/{filename}
```

Widths chosen for:
- **240w** → 160px collapsed card @ 1.5× DPR
- **480w** → ~300px carousel @ 1.5× DPR
- **960w** → desktop carousel + retina + expanded view on wide screens

All three generated by PIL in a single pass over the opened `Image` object (one decode, three `resize()` + `save()` calls). Maintain aspect ratio (`Image.thumbnail` with a `(width, 10000)` box) so portrait and landscape originals both look right — the current fixed-600×400 squash goes away.

The original upload is still stored unchanged at `/static/photos/rooms/{room_id}/{filename}` and remains the full-resolution source for the lightbox.

### 2. Backend — schema

In `agent_api/api/rooms/photo_schemas.py`, replace `thumbnail_url: str` with a single structured field:

```python
class PhotoResponse(BaseModel):
    id: int
    filename: str
    sort_order: int
    url: str  # full-resolution original (lightbox only)
    thumbnails: dict[int, str]  # { 240: "...", 480: "...", 960: "..." }
```

No legacy alias field. No backward-compat transitional state. `url` stays as the full-res source.

### 3. Frontend — hook type

In `client/src/hooks/usePhotos.ts`, mirror the schema change:

```ts
export interface PhotoResponse {
  id: number
  filename: string
  sort_order: number
  url: string
  thumbnails: Record<number, string>  // keyed by width in px
}
```

React Query caching stays as-is; no refetch logic changes.

### 4. Frontend — responsive `<img>` in every search surface

All card images emit `srcset` + `sizes` + lazy attributes + explicit dimensions:

```tsx
<img
  src={p.thumbnails[480]}
  srcSet={`${p.thumbnails[240]} 240w, ${p.thumbnails[480]} 480w, ${p.thumbnails[960]} 960w`}
  sizes="(min-width: 1024px) 300px, 160px"
  loading="lazy"
  decoding="async"
  width={600}
  height={400}
  alt={...}
/>
```

The browser picks the right file per DPR and viewport. `width`/`height` give an intrinsic aspect box before the fetch resolves — eliminates CLS.

**Eager-loading exception**: the thumbnail of the *first* card in the list should be `loading="eager" fetchPriority="high"` so above-the-fold LCP isn't delayed. Thread a `priority?: boolean` prop through from `DesignC.tsx` and set it on index 0.

### 5. Frontend — lazy-mount the expanded carousel

In `RoomCardVertical.tsx`, wrap the carousel `<img>` map in `{isExpanded && ...}`. The `grid-rows-[0fr→1fr]` height animation still works — grid-template-rows animates independently of mounted content, so mounting on expand just means the target row height is non-zero when content appears.

This is the single biggest bandwidth win: collapsed cards fetch zero carousel images. For a 16-room list, that's ~160 images eliminated from initial load.

Also swap the carousel to use the **480w thumbnail** (`thumbnails[480]`) instead of the full-res `url`. The lightbox remains the only surface that fetches the original.

### 6. Frontend — PhotoLightbox

`PhotoLightbox.tsx` already uses `photo.url` (full-res), which is correct. Add `loading="lazy"` to photos that aren't the initially-shown index so clicking one photo doesn't eagerly pull the whole album.

### 7. Frontend — other surfaces

Apply the same responsive img pattern to:
- `client/src/components/prototype/RoomCardCompact.tsx`
- `client/src/components/gen-ui-compopnents/RoomCard.tsx`
- `client/src/components/prototype/ResortMap.tsx` (map background image — single img, just add `loading="lazy"` + dimensions; no srcset needed since it's one fixed asset)

## Verification

1. **Clean state**: because there's no backfill, clear `agent_api/api/static/photos/rooms/*` before testing so no stale single-size thumbnails interfere.
2. **Upload path**: upload a photo via the existing admin UI → inspect `/static/photos/rooms/{id}/thumbnails/` → should see `240/`, `480/`, `960/` subdirectories, each containing the same filename, progressively larger.
3. **API response**: `GET /api/rooms/{id}/photos` → each item has `thumbnails: {240, 480, 960}` and `url`.
4. **Initial list load** (`/prototype`): DevTools → Network → Img filter.
   - Expect **~1 image per card** on first render (one 240w or 480w thumbnail each, depending on DPR).
   - Expect **zero carousel images** until a card is expanded.
   - Cards below the fold should defer their fetches (native `loading="lazy"`).
5. **Expand a card**: that room's carousel fetches 480w thumbnails.
6. **Open the lightbox from the carousel**: now (and only now) the full-res `url` fetches.
7. **DPR check**: on a 1× desktop viewport the collapsed card picks the 240w file; on Retina it picks 480w; on desktop carousel Retina it picks 960w. Verify in Network by looking at which URL was requested.
8. **CLS**: Lighthouse Performance audit on `/prototype` — CLS should be ~0, "Properly size images" and "Defer offscreen images" audits both pass, "Efficiently encode images" should pass for the 240/480 variants.
9. **LCP**: first card's thumbnail should be the LCP element and should be eager-loaded (no lazy delay).

## Out of Scope

- WebP / AVIF conversion — ~30% additional shrink but adds `<picture>` fallback complexity and Pillow needs `libwebp`. Defer.
- CDN / image proxy (imgproxy, Cloudflare Images) — worthwhile later, bigger architectural change.
- Blurhash / LQIP — shimmer is sufficient for now.
- Photo list pagination — only relevant if a room ever has > 30 photos.
